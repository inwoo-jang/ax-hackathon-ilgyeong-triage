## 1. 개요 (Executive Summary)

**한 줄 요약**
미래내일 일경험 사업 **운영자**가, 300명 규모 참여자의 8주 출결 데이터에서 **"지금 누구를 · 왜 · 무엇부터 챙겨야 하는지"** 를 로그인 직후 우선순위 큐(오늘 할 일)로 확인하고, AI가 만든 조치 문구를 검토·확정할 수 있게 하는 운영 트리아지 어드민.

**배경**
미래내일 일경험 사업(주관: 고용노동부 / 운영: 대한상공회의소)의 운영자는 다수 참여자의 출결·진도·서류·수당 상태를 매번 직접 확인해 "누구를 먼저 챙길지, 어떤 조치를 할지"를 수작업으로 판단한다. 단순 상태 분류는 누구나 하지만, **우선순위화(triage)와 조치 연결**이 담당자 머릿속에만 남아 재사용·표준화되지 않는다. 기존 도구(청년일경험 포털, 기관 자체 엑셀, 일반 LMS)는 신청·조회·집계까지만 다루고, 비어 있는 **"운영자 판단 레이어"** 를 채우는 것이 본 제품의 목적이다.

**범위 (Scope)**

| In-scope (이번 버전) | Out-of-scope (이번에 안 함) |
|---|---|
| 4메뉴: 오늘 할 일 / 참여자 관리 / 처리 내역 / 멘토별 현황 | 실데이터 연동·인증 시스템 (더미 300명 기반) |
| 개입가치(swing) 정렬 + AI 조치 문구 생성 | 참여자·기업 대상 기능 (운영자 전용) |
| 출석률 → 수당 구간·금액 환산 + 만회 분석·What-if | 자동 조치 확정·자동 발송 (항상 운영자 수동) |
| 출석시간 수정(mock)·연락 캘린더·처리 상태 관리 | 실제 수당 지급 처리·정산 연계 |
| 멘토별 현황 뷰(보조) | 모바일 앱 (데스크톱 웹 우선) |

## 2. 문제 정의

**사용자 문제 (현재 상태)**
운영자는 매번 수작업으로 다음 4가지를 판단해야 한다.
1. 누가 어떤 상태인지
2. 누구를 우선 판단해야 하는지
3. 어떤 걸 우선 확인해야 하는지
4. 누구에게 어떤 안내·조치가 필요한지

→ 300명 규모에서 이 판단을 사람이 반복하면 느리고, 담당자마다 기준이 달라 일관성이 없으며, 판단 노하우가 축적되지 않는다.

**근본 원인**
- 데이터는 흩어져 있고(출결·서류·수당 별도), 이를 종합해 "지금 독려하면 얼마가 달라지는가"를 보여주는 지표가 없다.
- 출석률을 단순 %로만 보고 **개입가치(swing, 최대출석 시 수당 − 현재 유지 시 수당)** 로 환산하지 않아, 개입 효율이 높은 대상을 놓친다.
- 상태 → 이유 → 조치가 끊겨 있어, 판단 결과가 표준화·재사용되지 않는다.

**기회**
- 운영 판단 시간을 단축하고, 담당자 간 판단 기준을 표준화한다.
- 출석 장려 여지가 큰 참여자(만회 가능군)에 우선 개입해 **평균 출석률**과 **4주 단위 지급금액**을 끌어올린다 (과제 핵심 KPI).

## 3. 목표 (Goals & Non-Goals)

**비즈니스 목표 (과제 출제 의도와 정렬)**
- 운영 판단 효율성 제고: 운영자가 로그인 직후 "오늘 먼저 챙길 사람"을 개입가치 순으로 본다.
- 출석률·지급금액 제고: 만회 가능군(출석 장려 대상) 우선 개입으로 평균 출석률과 지급액을 높인다.

**사용자 목표 (운영자)**
- 흩어진 상태를 한 화면에서 개입가치 순으로 본다.
- 각 대상의 **상태 → 이유 → 구체적 조치**를 하나의 흐름으로 확인·확정한다.
- 출석률을 금액(개입가치)으로 환산해 "지금 독려하면 결과가 크게 바뀌는 대상"을 먼저 처리한다.

**Non-Goals (이번에 의도적으로 안 하는 것)**
- AI 자동 확정 (최종 결정은 항상 운영자).
- 참여자/기업 대상 기능, 실제 수당 정산.
- 실시간 데이터 파이프라인 (해커톤은 더미 CSV 기반 데모).

## 5. 사용자 페르소나

**페르소나 1 — 운영 담당자 (Primary)**
- 이름/역할: 김운영, 32세, 수행기관 일경험 사업 운영 담당
- 상황: 멘토·서류·수당·출결을 엑셀과 자체 시스템으로 병행 관리. 담당 참여자 수십~수백 명.
- 목표: 매일 아침 "오늘 누구부터, 왜, 무엇을 챙길지" 빠르게 정하고 싶다.
- 불만: 데이터가 흩어져 매번 직접 대조해야 하고, 출석을 더 올릴 여지가 있는 사람을 놓쳐 지급액이 깎인다. 판단 기준이 매번 머릿속에서 새로 만들어진다.

**페르소나 2 — 멘토/슈퍼바이저 (Secondary)**
- 상황: 멘토 1명당 담당 그룹 운영. 자기 담당 그룹의 출석 장려·만회불가 신호를 먼저 보고 싶다.
- 목표: "내 담당자 중 출석 장려 대상·만회불가자"를 그룹 단위로 확인.

**사용 시나리오**
> 월요일 오전 9시, 김운영이 어드민에 로그인한다. "오늘 할 일" 상단에 개입가치 순 우선순위 큐가 떠 있다. 1순위는 "출석 장려 대상 — 지금 독려해 만땅 출석 시 직무훈련+일경험 합산 수당이 크게 회복되는 대상". 김운영은 상세 패널에서 만회 분석과 What-if로 효과를 확인하고, AI 조치 카드의 문자 문구를 검토·수정해 보낸 뒤 [연락 완료]로 처리한다. 처리 내역에 기록되고 다음 순위로 넘어간다.

## 6. 핵심 기능 요구사항

### F1. 오늘 할 일 (P0) — 승부처
- **설명**: 로그인 직후 랜딩. 코드가 계산한 결과를 **개입가치(swing) 정렬 우선순위 큐**로 노출하고, 상단에 2헤드라인과 요약 지표를 보여준다.
- **인수 조건**
  - 운영자가 로그인하면 **개입가치 내림차순**으로 정렬된 처리 대상 큐가 보인다.
  - 각 큐 항목에 대상 이름 · 개입가치 금액 · feasib 분류(출석 장려 대상/안전/만회불가) · 핵심 이유(한 줄) · 추천 조치 요약이 표시된다.
  - 큐 항목 클릭 시 해당 참여자 상세 패널로 이동한다.
  - 요약 지표가 현재 데이터 기준으로 정확히 집계된다(처리 대상 232명, 출석 장려 대상 229 / 안전 63 / 만회불가 8, 서류 미비 10).
  - **[2헤드라인]** 최상단에 핵심 금액 두 축을 큰 글씨로 노출한다: ① **오늘 개입으로 좌우되는 금액**(개입가치 합 ₩16,631,250) ② **서류 미비로 보류된 금액**(₩4,050,000). 건수 카드가 "얼마나 바쁜지"라면, 이 숫자는 "왜 지금 해야 하는지"를 보여준다.
  - **[정렬·필터]** 큐 정렬은 개입가치·출석률·가나다(↑↓ 방향 토글). 필터는 지급확정 주간(4·8주) 모아보기 / 연락처 보기 / 서류 미비만 보기.
  - **[종료 임박]** 코호트 종료가 가까운 대상(종료 D-n주 / 마지막주)을 표기한다. 정산은 4주 단위 2회차(P1=1~4주, P2=5~8주)로 진행 중인 기간만 고려한다.

### F2. 참여자 관리 (P0)
- **설명**: 검색·필터·정렬이 가능한 참여자 목록. 상태/수당 구간/수당 유형 필터와 일괄 문구 생성 제공.
- **인수 조건**
  - 이름 검색이 동작한다(이름은 실명 아님 · 마스킹하지 않음).
  - feasib 분류·수당구간·수당유형(직무훈련/일경험)·서류 미비 등으로 필터링된다.
  - 개입가치·출석률·가나다 등으로 정렬 가능하다(방향 토글).
  - **[동일 유형 일괄 처리]** 배치 체크박스로 선택한 대상에게 **AI가 각자 맥락(이름·남은 출석 시간·개입가치)에 맞춰 개별 안내 문구를 동시 생성**하고, 운영자 일괄 검토 후 한 번에 발송(mock)한다. 300명 규모에서 단건 반복 대신 스케일을 AI로 처리하는 핵심 동선.

### F3. 참여자 상세 패널 (P0)
- **설명**: 개인 패널. 판단 근거(출결·서류·수당)와 **출석률 → 수당 환산·만회 분석**을 표시하고, AI 조치 카드를 **검토/수정**할 수 있다.
- **인수 조건**
  - **[만회 분석]** 직무훈련·일경험 각각 현재 출석률과 도달 가능 범위(남은 주 0~만땅 기준 하한~상한)가 표시된다.
  - **[정산 기간별 출석률]** 1~4주/5~8주 현재 출석률과 상태(정산확정/진행중/미시작)가 보이고, 수당 구간 기준표를 (?) 토글로 확인한다.
  - **[What-if 시뮬레이터]** 주차별 남은 출석 시간을 입력하면 출석률 게이지와 예상 수당이 **실시간 재계산**된다. 기본값=현재 출석률 유지 시 시간, max 체크박스로 남은 주 만땅 시뮬.
  - **[출석시간 수정]** 주차별 출석 시간 보기 + 수정 모드. 현재 정산기간 주차만 수정 가능(P1은 🔒 잠금), 저장 시 서버 전송(mock)+만회 재계산.
  - **[연락 캘린더]** 6월 캘린더에서 날짜 클릭=기록 추가, 기록 클릭=수정/삭제. 처리 내역과 연동.
  - **[AI 조치 카드]** 상태 요약 / 우선순위 근거 / 📱메시지 문구 / 📞전화 스크립트 / ✏️추가 지시(상황·톤)+다시 생성.
  - **[하단 고정 액션바]** [연락 완료] [출석시간 수정] [보류], 4·8주는 [확정 완료] 추가. **자동 확정·자동 발송 없음.**

### F4. 처리 내역 (P1)
- **설명**: 처리 이력 추적(보조). 언제·누구에게·어떤 조치를 했는지 로그로 기록.
- **인수 조건**
  - 처리 결과가 시간순으로 기록·조회된다. 컬럼: 시각·참여자·조치·결과(확정완료/연락완료/보류).
  - 참여자별로 필터링해 이력을 볼 수 있고, 행 클릭 시 상세로 이동한다.
  - **[연락 캘린더 연동]** 상세 패널의 연락 기록과 처리 내역이 동기화된다.

### F5. 멘토별 현황 (P2)
- **설명**: 멘토(프로그램 그룹) 단위로 담당 그룹의 출석 장려·만회불가 신호를 묶어 보여준다.
- **인수 조건**: 멘토 기준으로 담당 참여자를 그룹핑해 그룹별 요약(feasib 분포·개입가치 합 등)을 표시한다.

### AI 활용 지점 (핵심 차별점)
> **역할 분리** — 계산은 전부 코드(규칙), AI는 표현만 재작성한다. 규칙이 만든 숫자·사실을 "기준 초안"으로 고정하고 AI는 표현만 바꿔 환각을 방지한다.
1. **상태 요약**: 코드가 만든 출석률·수당·개입가치·만회판정을 운영자가 읽기 좋게 요약.
2. **우선순위 근거**: 왜 이 순서로 떠 있는지(개입가치·만회 여부) 자연어 설명.
3. **조치 문구 생성**: 대상 상태에 맞는 문자·전화 안내 문구를 단건·일괄 생성하되, 운영자 검토·수정 후 발송(mock).
4. **피드백 루프**: 운영자의 수정·추가 지시(상황·톤)를 반영해 다시 생성.

## 7. 핵심 판단 로직 (수당 규정 반영)

수당은 4주 단위 출석률에 따라 차등 지급된다. 직무훈련·일경험 두 수당을 **별개로 계산해 합산**하며, 출석률 = Σ출석 ÷ Σ배정(구간 합). **지금 독려하면 더 받을 수 있는 금액(개입가치)이 큰 참여자**를 우선순위 상단에 둔다.

**직무훈련 수당 (월, 출석률 4주 단위)**

| 출석률 | 지급액 |
|---|---|
| 90%↑ | 150,000원 |
| 70~90% | 131,250원 |
| 60~70% | 112,500원 |
| 50~60% | 93,750원 |
| 50% 미만 | 미지급 |

**일경험 수당 (월, 출석률 4주 단위)**

| 출석률 | 지급액 |
|---|---|
| 90%↑ | 300,000원 |
| 70~90% | 262,500원 |
| 60~70% | 225,000원 |
| 50~60% | 187,500원 |
| 50% 미만 | 미지급 |

**만회 판정 (feasib) — 3분류**
- **만회불가**: 직무·일경험 둘 다 max(남은 주 만땅 시 상한)<50% → 다 채워도 미지급.
- **안전**: 둘 다 cur(이미 지난 주차의 확정 출석률)≥90% → 이미 잘하고 있음.
- **출석 장려 대상**: 그 외 — 아직 90% 미만, 출석을 올릴 여지가 있는 대상. (화면 표기는 "출석 장려 대상")
- **현재 분포**: 출석 장려 대상 229 / 안전 63 / 만회불가 8. 처리 대상(actionable = 출석 장려 대상 또는 서류 미비&만회 가능) 232명, 서류 미비 10명.

**우선순위 = 개입가치(swing) 정렬**
- `swing = max(0, 최대출석 시 수당 − 현재 출석률 유지 시 수당)`. cur이 없으면 직전 P1 습관 출석률로 대체.
- "지금 독려해 더 받을 수 있는 금액"으로, 큐 기본 정렬 기준이다.
- 보조 신호: 코호트 종료 임박(종료 D-n주/마지막주), 서류 미비.

## 8. 비기능 요구사항

- **성능**: 300명 규모 데이터 정렬·필터 결과가 체감 지연 없이(목표 1초 이내) 표시.

- **AI 신뢰성**: 우선순위·조치는 항상 **근거(이유)와 함께** 제시하며 운영자 검토를 전제로 한다(자동 확정 금지).
- **접근성/환경**: 데스크톱 웹 우선, 최신 브라우저 기준. 화면녹화용 단일 HTML 데모.
- **확장성(설계 고려)**: 수당 규정·구간 기준을 설정값으로 분리해 사업/연도 변경 시 로직 수정 없이 대응 가능하도록 지향.

### AI 연동 (★ 실서비스 vs 데모 구분)
- **실서비스**: 백엔드가 Anthropic Claude API(`POST /api/ai/recommend`)를 호출하고, **API 키는 서버에 보관**한다. → **Anthropic API 키 필수.**
- **데모 우회(3경로, 자동 폴백)** — 우상단에서 모드 선택, 실패 시 자동으로 mock 폴백:
  - ① 로컬 Claude 브리지: `bridge.py`가 설치된 Claude Code CLI(`claude`)를 호출, **키 불필요**.
  - ② Anthropic API 키 런타임 입력: 브라우저 직접 호출(`anthropic-dangerous-direct-browser-access` 헤더, claude-sonnet-4-6).
  - ③ mock: 규칙 기반 사전 생성, 키·인터넷 불필요로 녹화 안정.
- ⚠️ **①②는 데모/로컬 시연 전용 우회**다. 실서비스에서는 키를 서버에 두고 백엔드에서 호출한다.
