## 1. 개요 (Executive Summary)

한 줄로 요약하면, 미래내일 일경험 사업 운영자가 300명 규모 참여자의 8주 출결 데이터를 놓고 "지금 누구를, 왜, 무엇부터 챙겨야 하는지"를 로그인 직후 우선순위 큐(오늘 할 일)에서 바로 확인하고, AI가 만든 조치 문구를 검토·확정할 수 있게 하는 운영 트리아지 어드민이다.

**배경**

미래내일 일경험 사업(주관: 고용노동부 / 운영: 대한상공회의소)의 운영자는 여러 참여자의 출결·진도·서류·수당 상태를 매번 직접 확인하면서 "누구를 먼저 챙길지, 어떤 조치를 할지"를 수작업으로 판단한다. 상태를 단순히 분류하는 일은 누구나 할 수 있지만, 정작 우선순위를 매기고 조치까지 연결하는 판단은 담당자 머릿속에만 남아 재사용되거나 표준화되지 못한다. 청년일경험 포털, 기관 자체 엑셀, 일반 LMS 같은 기존 도구는 신청·조회·집계 단계까지만 커버한다. 그 위에 비어 있는 운영자 판단 레이어를 채우는 것이 이 제품의 목적이다.

**범위 (Scope)**

| 이번 버전에서 다루는 것 | 이번에는 다루지 않는 것 |
|---|---|
| 4메뉴: 오늘 할 일 / 참여자 관리 / 처리 내역 / 멘토별 현황 | 실데이터 연동·인증 시스템 (더미 300명 기반) |
| 개입가치(swing) 정렬과 AI 조치 문구 생성 | 참여자·기업 대상 기능 (운영자 전용) |
| 출석률을 수당 구간·금액으로 환산, 만회 분석·What-if | 자동 조치 확정·자동 발송 (조치는 항상 운영자 수동) |
| 출석시간 수정(mock)·연락 캘린더·처리 상태 관리 | 실제 수당 지급 처리·정산 연계 |
| 멘토별 현황 뷰(보조) | 모바일 앱 (데스크톱 웹 우선) |

## 2. 문제 정의

**사용자 문제 (현재 상태)**

운영자는 매번 수작업으로 네 가지를 판단해야 한다. 누가 어떤 상태인지, 누구를 먼저 판단해야 하는지, 무엇부터 확인해야 하는지, 그리고 누구에게 어떤 안내와 조치가 필요한지다.

300명 규모에서 이 판단을 사람이 반복하면 우선 느리다. 담당자마다 기준이 달라 결과가 들쭉날쭉하고, 그 과정에서 쌓인 판단 노하우도 어디에도 남지 않는다.

**근본 원인**

데이터가 출결·서류·수당으로 흩어져 있는데, 이를 묶어 "지금 독려하면 얼마가 달라지는가"를 보여주는 지표가 없다. 출석률을 단순 %로만 보고 개입가치(swing = 최대출석 시 수당 − 현재 유지 시 수당)로 환산하지 않으니, 정작 개입 효율이 높은 대상을 놓친다. 상태와 이유, 조치가 따로 노는 탓에 판단 결과가 표준화되거나 재사용되지도 않는다.

**기회**

운영 판단에 드는 시간을 줄이고 담당자 간 기준을 맞출 수 있다. 출석을 끌어올릴 여지가 큰 참여자(만회 가능군)에 먼저 개입하면 평균 출석률과 4주 단위 지급금액을 함께 높일 수 있는데, 이것이 과제의 핵심 KPI다.

## 3. 목표 (Goals & Non-Goals)

**비즈니스 목표 (과제 출제 의도와 정렬)**

과제 출제 의도는 크게 둘이다. 하나는 운영 판단 효율을 높이는 것으로, 운영자가 로그인하자마자 오늘 먼저 챙길 사람을 개입가치 순으로 보게 한다. 다른 하나는 출석률과 지급금액을 끌어올리는 것으로, 만회 가능군(출석 장려 대상)에 우선 개입해 평균 출석률과 지급액을 높인다.

**사용자 목표 (운영자)**

운영자는 흩어진 상태를 한 화면에서 개입가치 순으로 보고, 각 대상의 상태와 이유, 구체적인 조치를 하나의 흐름으로 확인·확정하고 싶어 한다. 출석률을 개입가치(금액)로 환산해 두면, 지금 독려했을 때 결과가 크게 바뀌는 대상부터 처리할 수 있다.

**Non-Goals (이번에 의도적으로 안 하는 것)**

AI가 조치를 자동으로 확정하는 일은 하지 않는다. 최종 결정은 언제나 운영자 몫이다. 참여자·기업 대상 기능과 실제 수당 정산도 범위 밖이며, 실시간 데이터 파이프라인 역시 다루지 않는다(해커톤은 더미 CSV 기반 데모다).

## 5. 사용자 페르소나

**페르소나 1 — 운영 담당자 (Primary)**

김운영, 32세, 수행기관에서 일경험 사업 운영을 담당한다. 멘토·서류·수당·출결을 엑셀과 자체 시스템에 나눠 관리하고, 맡은 참여자는 수십에서 수백 명에 이른다. 매일 아침 "오늘 누구부터, 왜, 무엇을 챙길지"를 빠르게 정하고 싶지만, 데이터가 흩어져 있어 매번 직접 대조해야 한다. 출석을 더 올릴 여지가 있는 사람을 놓쳐 지급액이 깎이는 일도 생기고, 판단 기준은 그때그때 머릿속에서 새로 만들어진다.

**페르소나 2 — 멘토/슈퍼바이저 (Secondary)**

멘토는 한 명당 담당 그룹을 운영한다. 자기 그룹의 출석 장려·만회불가 신호를 먼저 보고 싶어 하며, "내 담당자 중 출석 장려 대상과 만회불가자"를 그룹 단위로 확인하는 것이 목표다.

**사용 시나리오**

월요일 오전 9시, 김운영이 어드민에 로그인한다. "오늘 할 일" 상단에는 개입가치 순으로 정렬된 우선순위 큐가 떠 있다. 1순위는 출석 장려 대상, 즉 지금 독려해 만땅 출석을 채우면 직무훈련과 일경험 합산 수당이 크게 회복되는 대상이다. 김운영은 상세 패널에서 만회 분석과 What-if로 효과를 가늠하고, AI 조치 카드의 문자 문구를 검토·수정해 보낸 뒤 [연락 완료]로 처리한다. 결과는 처리 내역에 기록되고, 화면은 다음 순위로 넘어간다.

## 6. 핵심 기능 요구사항

### F1. 오늘 할 일 (P0) — 승부처

로그인 직후 보게 되는 랜딩 화면이다. 코드가 계산한 결과를 개입가치(swing) 순으로 정렬한 우선순위 큐로 노출하고, 상단에 2헤드라인과 요약 지표를 함께 띄운다.

인수 조건은 다음과 같다.

- 운영자가 로그인하면 개입가치 내림차순으로 정렬된 처리 대상 큐가 보인다.
- 각 큐 항목에는 대상 이름, 개입가치 금액, feasib 분류(출석 장려 대상/안전/만회불가), 핵심 이유 한 줄, 추천 조치 요약이 표시된다.
- 큐 항목을 클릭하면 해당 참여자 상세 패널로 넘어간다.
- 요약 지표가 현재 데이터 기준으로 정확히 집계된다(처리 대상 232명, 출석 장려 대상 229 / 안전 63 / 만회불가 8, 서류 미비 10).
- [2헤드라인] 최상단에 핵심 금액 두 축을 큰 글씨로 노출한다. 하나는 오늘 개입으로 좌우되는 금액(개입가치 합 ₩16,631,250)이고, 다른 하나는 서류 미비로 보류된 금액(₩4,050,000)이다. 건수 카드가 "얼마나 바쁜지"를 말한다면, 이 두 숫자는 "왜 지금 해야 하는지"를 말한다.
- [정렬·필터] 큐 정렬 기준은 개입가치, 출석률, 가나다이며 ↑↓ 방향을 토글한다. 필터로는 지급확정 주간(4·8주) 모아보기, 연락처 보기, 서류 미비만 보기를 제공한다.
- [종료 임박] 코호트 종료가 가까운 대상(종료 D-n주 / 마지막주)을 표기한다. 정산은 4주 단위 2회차(P1=1~4주, P2=5~8주)로, 진행 중인 기간만 고려한다.

### F2. 참여자 관리 (P0)

검색·필터·정렬이 되는 참여자 목록이다. 상태, 수당 구간, 수당 유형으로 거를 수 있고 일괄 문구 생성도 지원한다.

인수 조건은 다음과 같다.

- 이름 검색이 동작한다(이름은 실명이 아니며 마스킹하지 않는다).
- feasib 분류, 수당구간, 수당유형(직무훈련/일경험), 서류 미비 등으로 필터링된다.
- 개입가치, 출석률, 가나다 등으로 정렬할 수 있고 방향을 토글한다.
- [동일 유형 일괄 처리] 배치 체크박스로 대상을 고르면, AI가 각자의 맥락(이름·남은 출석 시간·개입가치)에 맞춰 개별 안내 문구를 한꺼번에 생성한다. 운영자가 일괄 검토한 뒤 한 번에 발송(mock)한다. 300명 규모에서 단건을 반복하는 대신 스케일을 AI로 처리하는 핵심 동선이다.

### F3. 참여자 상세 패널 (P0)

개인 단위 패널이다. 판단 근거인 출결·서류·수당을 보여주고, 출석률을 수당으로 환산한 만회 분석을 함께 띄운다. AI 조치 카드는 운영자가 검토하고 수정할 수 있다.

인수 조건은 다음과 같다.

- [만회 분석] 직무훈련과 일경험 각각에 대해 현재 출석률과 도달 가능 범위(남은 주 0~만땅 기준 하한~상한)를 보여준다.
- [정산 기간별 출석률] 1~4주와 5~8주의 현재 출석률, 상태(정산확정/진행중/미시작)가 보이고, 수당 구간 기준표는 (?) 토글로 펼쳐 본다.
- [What-if 시뮬레이터] 주차별 남은 출석 시간을 입력하면 출석률 게이지와 예상 수당이 실시간으로 다시 계산된다. 기본값은 현재 출석률 유지 시 시간이고, max 체크박스로 남은 주를 만땅으로 채운 경우를 시뮬레이션한다.
- [출석시간 수정] 주차별 출석 시간을 보고 수정 모드로 들어간다. 현재 정산기간 주차만 수정할 수 있고(P1은 잠금) 저장하면 서버로 전송(mock)된 뒤 만회가 다시 계산된다.
- [연락 캘린더] 6월 캘린더에서 날짜를 클릭하면 기록을 추가하고, 기록을 클릭하면 수정·삭제한다. 처리 내역과 연동된다.
- [AI 조치 카드] 상태 요약, 우선순위 근거, 메시지 문구, 전화 스크립트, 그리고 상황·톤을 더한 추가 지시 후 다시 생성까지 담는다.
- [하단 고정 액션바] [연락 완료] [출석시간 수정] [보류]가 있고, 4·8주에는 [확정 완료]가 추가된다. 자동 확정도 자동 발송도 없다.

### F4. 처리 내역 (P1)

처리 이력을 추적하는 보조 화면이다. 언제 누구에게 어떤 조치를 했는지 로그로 남긴다.

인수 조건은 다음과 같다.

- 처리 결과가 시간순으로 기록·조회된다. 컬럼은 시각, 참여자, 조치, 결과(확정완료/연락완료/보류)다.
- 참여자별로 필터링해 이력을 볼 수 있고, 행을 클릭하면 상세로 이동한다.
- [연락 캘린더 연동] 상세 패널의 연락 기록과 처리 내역이 서로 동기화된다.

### F5. 멘토별 현황 (P2)

멘토(프로그램 그룹) 단위로 담당 그룹의 출석 장려·만회불가 신호를 묶어 보여준다. 멘토 기준으로 참여자를 그룹핑해 그룹별 요약(feasib 분포, 개입가치 합 등)을 표시하는 것이 인수 조건이다.

### AI 활용 지점 (핵심 차별점)

핵심은 역할 분리다. 계산은 전부 코드(규칙)가 맡고, AI는 표현만 다시 쓴다. 규칙이 만든 숫자와 사실을 기준 초안으로 고정해 두고 AI는 문장만 바꾸기 때문에, 숫자가 흔들리거나 없는 사실이 끼어드는 환각을 막을 수 있다. AI는 네 군데에서 쓰인다.

1. 상태 요약: 코드가 만든 출석률·수당·개입가치·만회판정을 운영자가 읽기 좋게 다듬는다.
2. 우선순위 근거: 왜 이 순서로 떠 있는지(개입가치, 만회 여부)를 자연어로 설명한다.
3. 조치 문구 생성: 대상 상태에 맞는 문자·전화 안내 문구를 단건 또는 일괄로 만들되, 운영자 검토·수정을 거쳐 발송(mock)한다.
4. 피드백 루프: 운영자가 더한 수정과 추가 지시(상황·톤)를 반영해 다시 생성한다.

## 7. 핵심 판단 로직 (수당 규정 반영)

수당은 4주 단위 출석률에 따라 차등 지급된다. 직무훈련과 일경험 두 수당은 따로 계산한 뒤 합산하고, 출석률은 Σ출석 ÷ Σ배정(구간 합)으로 구한다. 우선순위 상단에는 지금 독려하면 더 받을 수 있는 금액, 곧 개입가치가 큰 참여자를 둔다.

**직무훈련 수당 (월, 출석률 4주 단위)**

| 출석률 | 지급액 |
|---|---|
| 90%↑ | 150,000원 |
| 70~90% | 131,250원 |
| 60~70% | 112,500원 |
| 50~60% | 93,750원 |
| 50% 미만 | 미지급 |

**일경험 수당 (월, 출석률 4주 단위)**

| 출석률 | 지급액 |
|---|---|
| 90%↑ | 300,000원 |
| 70~90% | 262,500원 |
| 60~70% | 225,000원 |
| 50~60% | 187,500원 |
| 50% 미만 | 미지급 |

**만회 판정 (feasib) — 3분류**

판정은 세 갈래다. 만회불가는 직무·일경험 둘 다 max(남은 주 만땅 시 상한)가 50% 미만인 경우로, 다 채워도 미지급이다. 안전은 둘 다 cur(이미 지난 주차의 확정 출석률)이 90% 이상인 경우로, 이미 잘하고 있는 대상이다. 나머지는 모두 출석 장려 대상인데, 아직 90%에 못 미쳐 출석을 올릴 여지가 있는 사람들이다(화면 표기도 "출석 장려 대상").

현재 분포는 출석 장려 대상 229 / 안전 63 / 만회불가 8이다. 처리 대상(actionable = 출석 장려 대상 또는 서류 미비 & 만회 가능)은 232명, 서류 미비는 10명이다.

**우선순위 = 개입가치(swing) 정렬**

개입가치는 `swing = max(0, 최대출석 시 수당 − 현재 출석률 유지 시 수당)`로 정의한다. cur이 없으면 직전 P1 습관 출석률로 대체한다. 말 그대로 지금 독려해 더 받을 수 있는 금액이며, 큐의 기본 정렬 기준이다. 코호트 종료 임박(종료 D-n주/마지막주)과 서류 미비는 보조 신호로 함께 본다.

## 8. 비기능 요구사항

- 성능: 300명 규모 데이터의 정렬·필터 결과가 체감 지연 없이 뜬다(목표 1초 이내).
- 데이터: 더미 300명(8주 주차별 출석/배정 시간)과 운영 판단 규정 요약본을 입력으로 쓴다. 실연동은 없다.
- AI 신뢰성: 우선순위와 조치는 늘 근거(이유)와 함께 제시하고, 운영자 검토를 전제로 한다. 자동 확정은 하지 않는다.
- 접근성/환경: 데스크톱 웹을 우선하고 최신 브라우저를 기준으로 한다. 화면녹화용 단일 HTML 데모다.
- 확장성(설계 고려): 수당 규정과 구간 기준을 설정값으로 분리해, 사업이나 연도가 바뀌어도 로직을 고치지 않고 대응할 수 있게 한다.

### AI 연동 (실서비스와 데모 구분)

실서비스에서는 백엔드가 Anthropic Claude API(`POST /api/ai/recommend`)를 호출하고, API 키는 서버에 보관한다. 즉 Anthropic API 키가 반드시 필요하다.

데모에서는 세 경로를 두고 자동으로 폴백한다. 우상단에서 모드를 고르며, 실패하면 자동으로 mock으로 떨어진다.

- 로컬 Claude 브리지: `bridge.py`가 설치된 Claude Code CLI(`claude`)를 호출한다. 키가 필요 없다.
- Anthropic API 키 런타임 입력: 브라우저에서 직접 호출한다(`anthropic-dangerous-direct-browser-access` 헤더, claude-sonnet-4-6).
- mock: 규칙 기반으로 미리 생성해 둔다. 키도 인터넷도 필요 없어 녹화가 안정적이다.

앞의 두 경로는 어디까지나 데모·로컬 시연용 우회다. 실서비스에서는 키를 서버에 두고 백엔드에서 호출한다.

## 9. 제약사항 및 가정

**제약**

해커톤 데모라 더미 CSV를 기반으로 하고, 인증과 실데이터 연동은 구현하지 않았다. 수당 구간과 지급액은 제공된 규정 요약본을 정답 기준으로 삼는다.

**가정 (검증 필요)**

사실로 확인된 것은, 기관마다 운영·접수·신청 절차가 다르고 그 절차를 모두 수행기관이 전담하며 중앙 포털은 신청·조회 중심이라는 점이다(출처: bizinfo 모집공고, 청년일경험 포털 yw.work24.go.kr). 반면 각 운영기관이 출결·수당 판단을 엑셀이나 자체 시스템으로 제각각 관리한다는 것은 추론에 가깝다. 직접 증거를 확인하지 못했으므로 현장 인터뷰와 실물 화면으로 검증해야 한다. 발표에서는 사실과 추론을 구분해 전달한다.
