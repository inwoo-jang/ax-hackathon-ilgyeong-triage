### 1-2. 런타임 파생값 (`derive(p)` 가 부여 · 출석시간 수정 시 재호출)

| 필드 | 타입 | 산출 | 의미 |

| `jc` / `wc` | object | `kcalc(p,'jt')` / `kcalc(p,'we')` | 직무훈련·일경험 만회 계산 결과 |

| ↳ `max` | float 0..1 | 남은 주 다 채울 때 상한 | |
| ↳ `min` | float 0..1 | 남은 주 0일 때 하한 | |
| ↳ `cAtt/cTot/remTot/pTot` | int | 확정 출석/확정 배정/남은 배정/기간 총배정 | 보조값 |

| `maxPay` | int | `payOf(jc.max)+payOf(wc.max)` | 최대 출석 시 수당 |
| `minPay` | int | `payOf(jc.min)+payOf(wc.min)` | 최소(남은 0) 수당 |
| `swing` | int | `max(0, maxPay - expPay)` | 개입가치. 지금 독려하면 더 받게 되는 금액 |
| `feasib` | `'결정적'\|'안전'\|'만회불가'` | 1-2 규칙 | 만회 3분류(화면 표기 "출석 장려 대상") |
| `actionable` | bool | `feasib==='결정적' \|\| (doc==='미비' && feasib!=='만회불가')` | 처리 대상 |
| `status` | `'untouched'\|'contacted'\|'paid'\|'hold'` | 초기 `'untouched'` | 처리 상태 |

## 2. 비즈니스 로직 (계산식 · 전부 규칙/코드)

### 2-1. 출석률 = Σ출석 ÷ Σ배정 (구간 합)
```js
fullRate(p,k,s,e)   = Σ_{i=s..e} p[k+'_w'][i] / Σ_{i=s..e} p[k+'_wt'][i]   // 전체 구간
periodConf(p,k,s,e) = 위와 동일하되 i ≤ cw-2 (현재 주차 이전=확정)만 합산 → 없으면 null
```
주차 개수가 아니라 시간 단위로 구간을 합산한다. 주마다 배정 시간이 다르기 때문에 사실상 가중 평균이 된다.

### 2-2. 정산 기간 & 수당 구간
- 4주 2회차: `P1 = 1~4주(idx 0~3)`, `P2 = 5~8주(idx 4~7)`. `curPeriod(cw)` → `cw≤4 ? [0,3] : [4,7]`.
- 구간 임계 `THR = [0.90, 0.70, 0.60, 0.50]`, 라벨 `['90%+','70-90%','60-70%','50-60%','<50% 미지급']`.
- `tierOf(r)`: r≥0.90→0 … r<0.50→4. `payOf(r,k)=PAY[k][tier][1]`.

| 4주 출석률 | 직무훈련(jt) | 일경험(we) |
|---|---|---|
| 90%+ | 150,000 | 300,000 |
| 70–90% | 131,250 | 262,500 |
| 60–70% | 112,500 | 225,000 |
| 50–60% | 93,750 | 187,500 |
| <50% | 0 (미지급) | 0 (미지급) |

만점 총수당 `FULL = 450,000`.

### 2-3. kcalc (현재 기간 만회 계산)
```js
// 현재 주차가 속한 기간 [s,e] 내에서
// i ≤ cw-2 : 확정(지난 주차)  →  cAtt, cTot 누적
// i  > cw-2: 남은 주차        →  remTot 누적
cur = cTot>0 ? cAtt/cTot : null           // 확정 출석률(데이터 없으면 null)
max = pTot ? (cAtt + remTot) / pTot : 0    // 남은 주 배정 만땅
min = pTot ? cAtt / pTot : 0               // 남은 주 0
```

### 2-4. feasib 3분류
| 분류 | 규칙 | 화면 표기 |
|---|---|---|
| 만회불가 | `jc.max < 0.50 && wc.max < 0.50` (둘 다 다 채워도 미지급) | "만회 불가" |
| 안전 | `jc.cur≥0.90 && wc.cur≥0.90` (둘 다 이미 90%+) | "이상 없음" |
| 결정적 | 그 외 (90% 미만, 올릴 여지) | "출석 장려" |

우선순위는 만회불가 < 안전 < 결정적 순서이며, 결정적부터 처리한다.

### 2-5. 개입가치 swing & 처리 대상 actionable
```
swing = max(0, maxPay - expPay)        // expPay의 cur가 null이면 직전 P1 습관율로 대체
actionable = (feasib==='결정적') || (doc==='미비' && feasib!=='만회불가')
```

### 2-6. 헤드라인 집계 (`HEAD`)
| 키 | 산식 | 값(현재 분포) |
|---|---|---|
| swingTotal | Σ swing over actionable | ₩16,631,250 |
| docHold | Σ expPay over (doc='미비' & feasib≠'만회불가') | ₩4,050,000 |
| decisiveCnt / safeCnt / failCnt | feasib 카운트 | 229 / 63 / 8 |
| docMiss | doc='미비' 카운트 | 10 |
| total | 전체 | 300 |

처리 대상(actionable)은 232명이다.

### 2-7. What-if 재계산 (클라이언트, 규칙 산식 재사용)
```
남은 주 i 에 입력시간 WFh[k][i] 누적 →
finalRate(k) = (cAtt + Σ WFh[k][i]) / pTot
simPay = payOf(jr,'jt') + payOf(wr,'we');   chg = simPay - expPay
```
입력은 `0 ≤ v ≤ 배정시간[i]` 범위로 클램프한다. 원본 데이터는 건드리지 않고 시뮬레이션 값만 화면에 표시한다.

## 3. 화면별 기능 명세

> 메뉴는 네 개다. 오늘 할 일(`dash`), 참여자 관리(`list`), 처리 내역(`logs`), 멘토별 현황(`mentors`). 라우팅은 `go(v)`가 담당한다.

### F1. 오늘 할 일 (`renderDash`)

화면은 위에서부터 헤드라인 카드 2개, 요약 카드 4개, 정렬·필터 바, 우선순위 큐 순으로 쌓인다.

- 헤드라인 ① 오늘 개입으로 좌우되는 금액 = `HEAD.swingTotal`(₩16,631,250). `?` 토글로 산식을 펼친다.
- 헤드라인 ② 서류 미비로 보류된 금액 = `HEAD.docHold`(₩4,050,000).
- 요약 카드: 전체 300 / 출석 장려 대상 229 / 서류 미비 10 / 만회 불가 8.
- 큐 대상(liveQ)은 필터 상태에 따라 달라진다.
  - 기본은 `status==='untouched' && actionable`.
  - 지급확정 주간 필터 ON이면 `(cw===4||cw===8) && status∉{paid,hold}`.
  - 서류 미비만 필터 ON이면 여기에 `doc==='미비'` 조건을 더한다.
- 정렬(QSORT)은 `swing`(개입가치, 기본), `rate`(출석률 = jc.cur+wc.cur), `name`(가나다) 세 가지다. 같은 버튼을 다시 누르면 방향이 토글된다(`QDIR` ↑↓).
- 필터: 지급확정 주간(4·8주), 연락처 보기(SHOWCONTACT), 서류 미비만 보기.
- 큐 카드에 들어가는 항목은 순위, 이름·성별·생년월일, 프로그램·현재 N주차·PN칩, feasib 칩과 근거 한 줄, 현재 출석률(jt/we), 연락처(연락처 보기 켜진 경우), 개입가치·최대 수당, `종료 D-n주` 배지와 상태칩이다.

**인수조건**
- Given 처리 대상이 0건, When `dash` 진입, Then "오늘 먼저 볼 대상이 없습니다." 빈 상태.
- Given 큐가 swing 정렬, When 개입가치 버튼 재클릭, Then 오름/내림 토글되고 화살표 표기 반전.
- Given 어떤 참여자 `status='paid'`, When 기본 큐 렌더, Then 큐에서 제외.

### F2. 참여자 관리 (`renderList` · `filtered` · `fillList`)

- 검색은 이름 부분일치로 동작한다(`nm.includes(q)`).
- 필터는 AND 결합이다. 수당유형(`pg.includes('직무훈련'|'일경험')`), 상태(feasib: 결정적/만회불가/안전), 서류(미비/완료)를 함께 건다.
- 정렬은 세 가지다. 우선순위순(`prioritySort`은 feasib, swing↓, 마감 임박, 서류미비 순으로 비교), 개입가치순, 주차순.
- 행에는 체크박스, 이름·성별·생년월일, 프로그램, 현재 주차·PN·D-n주, 출석률(jt/we), 최대 도달 수당, 개입가치, 상태칩, 서류칩, 처리상태칩이 들어간다.
- 일괄 처리(batch)는 다음 흐름으로 진행된다. 행을 체크하면 `BATCH` Set에 담기고 하단 배치바가 뜬다. `openBatch()` 모달에서 선택한 N명 각각에 대해 맥락에 맞는 개별 문구(genAI)를 한 번에 생성하고, `sendBatch()`로 일괄 발송(mock)한다. 발송된 건은 모두 `status='contacted'`로 바뀌고 LOGS에 기록된다.

**인수조건**
- Given 필터 결과 0건, When 렌더, Then "조건에 맞는 참여자 없음 · 필터를 초기화해 주세요".
- Given 3명 체크, When AI 일괄 문구 생성, Then 3개 카드 각각 이름·주차·개입가치 반영 문구 생성.
- Given 일괄 발송, When 확정, Then 미처리였던 대상만 `contacted`로 전이·로그 N건 추가.

### F3. 처리 내역 (`renderLogs`)
- 컬럼은 시각, 참여자, 조치, 결과 순이다. 결과값은 `paid`를 확정완료, `contacted`를 연락완료, `hold`를 보류로 표시한다.
- 행을 클릭하면 해당 참여자 상세(`openDetail`)가 열린다.
- 하단 요약에 확정완료 N, 연락완료 N, 보류 N을 집계한다.
- 빈 상태 문구는 "아직 확정된 조치가 없습니다."

### F4. 멘토별 현황 (`renderMentors`)
- 프로그램(pg) 10종이 곧 멘토 그룹이다. 그룹 카드에는 담당 인원, 현재 주차·종료 D-n주, 처리 대상 수, 처리 대상 상위 6명 칩(이름·성별·개입가치)을 담고, 칩을 누르면 상세가 열린다.

| 주차별 출석 시간 | 보기(`weeklyTable`)/스파크라인(`spark`). 수정 모드(`weeklyEdit`)에서는 현재 기간 주차만 다루며 P1은 잠금. 저장하면 서버 전송(mock)과 `derive` 재계산 | `editWeekly`,`saveWeekly`,`cancelEdit` |
| 연락 기록 | 6월 캘린더. 날짜를 누르면 그날 기록을 추가하고, 기록을 누르면 수정/삭제한다. 처리 내역(LOGS)과 연동 | `contactCal`,`addContactOn`,`editContact`,`saveContact`,`delContact` |
| AI 조치 카드 | 상태 요약, 우선순위 근거, 메시지, 전화 스크립트, 추가 지시(상황·톤), 다시 생성 | `genAI`,`regen`,`callClaude` |
| 하단 고정 액션바 | `[연락 완료][출석시간 수정][보류]`, 4·8주에는 `[확정 완료]`가 추가된다 | `confirmAction` |

**상세 인수조건**
- Given `p2stage`(cw>4), When 출석시간 수정 모드, Then P1(0~3주)은 잠금·읽기전용이고 P2 주차만 입력 가능.
- Given 출석시간 저장, When `saveWeekly`, Then 입력값 클램프 후 `derive(p)` 재계산 → 만회·수당·swing 갱신, toast "서버 전송 완료 (mock)".
- Given What-if "최대 출석 가정" 체크, When `setMax(true)`, Then 남은 주 입력칸이 배정시간 만땅으로 채워지고 수당 상한 표시.
- Given 캘린더 6/N 클릭, When `addContactOn(N)`, Then `contacts`에 추가 + 미처리면 `contacted` 전이 + LOGS에 "연락" 1건.
- Given cw가 4 또는 8, When 상세 열림, Then 액션바에 `확정 완료` 노출(그 외 미노출).

## 4. 상태 머신

```
            연락 완료 (contacted)
           ↗
미처리 ──────→ 확정 완료 (paid)   [4·8주만 가능]
(untouched) ↘
            보류 (hold)  →  3일 후 스누즈 재알림
```

| 액션 | 전이 | 로그 result | 비고 |
|---|---|---|---|
| 연락 완료 | untouched → `contacted` | `contacted`(연락완료) | contacts에 그날 기록 push |
| 확정 완료 | untouched → `paid` | `paid`(확정완료) | cw∈{4,8} 한정 |
| 보류 | * → `hold` | `hold`(보류) | 큐에서 빠지고 스누즈 |
| 캘린더/일괄 연락 | untouched → `contacted` | `contacted` | 자동 전이 |

- '처리중' 상태는 두지 않는다. 처리 내역 결과값은 확정완료/연락완료/보류 세 가지뿐이다.

## 5. API 명세

### 5-1. POST /api/ai/recommend (단건 AI 조치 문구)
실서비스에서는 백엔드가 Anthropic API를 호출하며 키는 서버에 보관한다. 계산값은 규칙이 만든 "기준 초안"으로 고정되고, AI는 표현만 다시 쓴다.

**Request**
```json
{
  "participant": { "nm":"이서연", "cw":3, "feasib":"결정적", "doc":"미비",
    "jc":{"cur":0.95,"max":0.97,"min":0.40}, "wc":{"cur":1.0,"max":1.0,"min":0.6},
    "swing":131250, "maxPay":450000, "expPay":318750 },
  "draft": { "msg":"규칙이 만든 문자 초안", "call":"규칙이 만든 전화 스크립트" },
  "instruction": "따뜻한 톤으로 / 장학금 안내 함께"
}
```
**Response** `200`
```json
{ "summary":"상태 요약(규칙 숫자 그대로)", "call":"전화 스크립트",
  "msg":"문자 문구", "reason":"우선순위 근거" }
```
> system 프롬프트가 "숫자·사실은 절대 바꾸지 말 것"을 강제한다. 응답 JSON은 `parseAI`가 `{...}` 부분만 추출하며, 추출에 실패하면 draft로 폴백한다.

**에러표**

| 코드 | 상황 | 클라이언트 처리 |
|---|---|---|
| 400 | participant/draft 누락 | 요청 중단, 토스트 |
| 401 | API 키 무효 | mock 폴백 |
| 429 | 레이트리밋 | mock 폴백 + 재시도 안내 |
| 5xx / timeout | 모델·네트워크 오류 | mock(genAI) 자동 폴백 |
| JSON 파싱 실패 | 비정형 응답 | draft(규칙 초안) 유지 |

### 5-2. PUT /api/participants/{id} (출석시간 수정 저장)
**Request**
```json
{ "jt_w":[11,10,10,12,9,11,0,0], "we_w":[8,10,9,8,9,9,0,0] }
```
- 제약: `0 ≤ jt_w[i] ≤ jt_wt[i]`를 서버가 검증하고, 초과하면 클램프한다. P2단계에서 P1(0~3주)을 수정하려 하면 거부한다(409 Locked).
**Response** `200` `{ "ok":true }`. 응답을 받으면 클라이언트가 `derive()`를 다시 돌린다.
> 데모에서는 `saveWeekly()`가 로컬 배열을 수정하고 `derive`를 재계산하는 mock으로 처리한다("서버 전송 완료 (mock)").

### 5-3. 데모 우회 3경로 (AI 호출, 자동 폴백)

| 경로 | 호출 방식 | 키 | 모델 | 비고 |
|---|---|---|---|---|
| ① 로컬 Claude 브리지 | `POST http://localhost:8787`로 보내면 `bridge.py`가 설치된 `claude` CLI를 실행 | 불필요 | claude-cli 인증 | 시연 권장. health=`GET /` |
| ② 브라우저 직접 호출 | `fetch https://api.anthropic.com/v1/messages` (헤더 `anthropic-dangerous-direct-browser-access:true`) | 런타임 입력 | `claude-sonnet-4-6` | 데모 전용 우회 |
| ③ mock | `genAI(p, prompt)` 규칙 기반 사전 생성 | 불필요 | — | 인터넷 불필요, 녹화 안정 |

`regen`은 다음 순서로 경로를 고른다. APIKEY가 있으면 ②, 없고 BRIDGE_OK이면 ①, 둘 다 실패하면 ③ mock으로 자동 폴백한다. `bridge.py`와 직접 호출은 데모 전용이며, 실서비스 경로는 §5-1을 따른다.

## 6. 환각 방지 · 경계 · 예외 처리

| 항목 | 규칙 |
|---|---|
| 숫자 고정 | 출석률·수당구간·금액·개입가치·만회판정·What-if는 모두 코드가 계산한다. AI는 그 값을 읽고 표현만 다시 쓴다. system 프롬프트에 "숫자·사실 변경 금지"를 명시한다 |
| AI 표현만 | `genAI`가 만든 summary/reason/msg/call이 "기준 초안"이다. `regen`은 톤·상황(instruction)만 반영한다 |
| 호출 실패 폴백 | API·브리지 호출이나 JSON 파싱이 실패하면 mock(genAI) 또는 draft로 자동 폴백하고 토스트로 알린다 |
| P1 수정 잠금 | P2단계(cw>4)에서 P1(0~3주)은 정산이 확정된 구간이라 읽기전용이다. 저장 루프가 P1 칸을 만들지 않는다 |
| What-if 클램프 | 입력은 `0 ≤ v ≤ 배정시간`으로 제한하고, 원본은 그대로 둔 채 시뮬값만 보여준다 |
| 현재 기간만 고려 | 5주차 이상이면 P1은 정산 확정이라 P2단계 서류는 강제 완료 처리하고, 만회는 현재 진행 기간만 판단한다 |
| 빈 상태 | 큐 0건, 리스트 0건, 로그 0건일 때 각각 안내 메시지를 띄운다 |
| 자동 확정·발송 금지 | 확정·연락·보류는 모두 운영자가 클릭해야 일어난다. AI는 초안만 만든다 |
